# LLM_DRIVEN_DESIGN.md - 为什么这个系统必须由 LLM 驱动

## 结论

这个系统的主要价值，不在“收集信息”，也不在“转发信息”。

主要价值在于：

**LLM 能基于用户资产、用户偏好、用户约束、市场新闻、轻量脚手架和近期变化，给出个人投资建议。**

如果没有这一层，系统最多只是一个更好看的信息搬运器，或者一个结构更复杂的市场简报器。

---

## 为什么不能把规则做得太满

当规则系统做得过满时，会发生几件坏事：

1. 程序先把结论写死
2. LLM 只能复述中间层输出
3. 用户资产被压缩成几个标签
4. 用户偏好和约束进不去主分析链
5. 建议退化成“栏目更整齐的市场解读”
6. 最终输出看起来像格式更好的文本，而不是更好的判断

这会直接削弱系统最有价值的能力。

---

## 正确分工

### 程序应该做的

- 读取和校验资产
- 持久化金融记忆
- 保留用户确认边界
- 拉取新闻与基础行情
- 生成最小市场状态与组合映射
- 维护必要的快照与日志
- 输出兜底与可追踪路径

### LLM 应该做的

- 理解组合结构
- 理解偏好与约束是否和当前结构匹配
- 综合当天市场变化和资产结构关系
- 判断哪些暴露更受益、哪些承压、哪些更值得买/卖/观察
- 在不确定时讲清边界，而不是假装确定
- 生成用户真正关心的个人投资建议

也就是说：

- **程序负责可靠性**
- **LLM 负责判断力**

---

## 纠偏后的主张

后续所有文档、代码和 prompt 都应遵守以下主张：

1. 宽输入，窄输出
2. 结构化数据是脚手架，不是最终答案
3. LLM 是主引擎，不是润色器
4. 用户资产、偏好与约束优先于机械市场播报
5. 健壮性必须保留，但不能以牺牲 LLM 分析空间为代价
6. 系统目标是给个人投资建议，不是生成更像样的市场简报

---

## 工程上的落实方式

为了既发挥 LLM，又保证系统健壮，当前应坚持：

- 原始上下文与结构化脚手架同时喂给 LLM
- 中间层输出候选解释，而不是定稿
- 用户约束应逐步进入主分析链
- Prompt 明确鼓励综合判断，明确禁止编造
- 渠道投递与分析目标解耦

---

## 判断标准

以后评估系统是否走偏，可以问一个简单问题：

> 如果把 LLM 拔掉，这个系统还剩下多少“投资建议”价值？

如果答案是“差不多还能跑”，那说明 LLM 还没有真正成为主驱动。

而这个项目追求的状态应该是：

> 没有 LLM，就没有这套 personal investment advisor 最核心的价值。
