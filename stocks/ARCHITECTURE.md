# ARCHITECTURE.md - Personal Investment Advisor Architecture

## 总原则

系统必须同时满足两件事：

1. **LLM 要有足够大的分析空间**
2. **工程层必须提供足够稳的输入与边界**

因此当前架构原则不是“规则优先，LLM 点缀”，而是：

**程序负责可靠输入、记忆边界和最小脚手架，LLM 负责生成个人投资建议。**

不要让程序过早替 LLM 写完答案，也不要把系统继续做成“个人市场简报器”。

---

## Layer 0：金融记忆层

职责：

- 保存用户长期金融资产
- 保存用户组合偏好、风险倾向与约束
- 明确哪些信息由用户确认、哪些只是备注

特点：

- 这是长期稳定层
- 不被短期市场信息覆盖
- 不允许 LLM 擅自改写

---

## Layer 1：外部市场输入层

职责：

- 提供新闻输入
- 提供基础行情输入
- 提供当天最小跨资产观察

特点：

- 这是临时上下文层
- 主要回答“今天市场发生了什么、哪里强、哪里弱、哪里分化”
- 这些输入服务于建议生成，不直接替用户做最终决策

---

## Layer 2：轻量脚手架层

职责：

- 对市场状态做轻量归纳
- 对用户组合做轻量映射
- 给 LLM 提供最低限度的稳定信号

注意：

- 这一层是脚手架，不是决策引擎
- 不应继续无限膨胀更多中间层
- 如果完整上下文支持更好的综合判断，LLM 应该有能力覆盖中间信号

---

## Layer 3：LLM 建议生成层（主引擎）

职责：

- 同时读取用户资产、偏好、约束、新闻、行情与轻量脚手架
- 综合判断当前市场对用户组合意味着什么
- 给出个人投资建议

这里的建议应包括：

- 当前市场判断
- 对用户组合的影响
- 哪些暴露更受益 / 更承压
- 哪些方向更值得关注
- 该买什么、该卖什么、该观察什么的方向性建议

这是整个系统最有价值的一层，也是主引擎。

如果这一层被压缩成模板润色器，那么整个系统就退化成信息转发器。

---

## Layer 4：投递与交互层

职责：

- OpenClaw 对话交互
- OpenClaw cron 调度
- Feishu 直发

要求：

- 建议生成与渠道投递解耦
- 用户资产更新需要确认边界
- 投递层不反向定义分析目标

---

## 健壮性原则

强调 LLM 驱动，并不意味着放弃工程性。

系统必须同时具备：

- 明确的数据来源边界
- 用户确认的资产更新机制
- 日志与失败兜底
- 最小可解释脚手架
- 关键链路可测试
- 投递链路稳定

工程层的任务，是让 LLM 在稳定地基上给出更好的个人投资建议，而不是把建议先写死。

---

## 当前设计边界

允许扩展：

- 更完整的用户约束信息
- 更可靠的市场输入
- 更好的建议质量
- 更顺滑的资产维护体验

不允许走偏：

- 不把系统做成重型投顾平台
- 不把短期新闻写入长期金融记忆
- 不让程序把 LLM 压成模板润色器
- 不为未来平台化继续横向扩张
